# ADR 001: プロジェクトの初期アーキテクチャ選定とNDA（機密保持）ポリシーの策定

- **Date:** 2026-05-26
- **Status:** Accepted
- **Context (背景):**
  本プロジェクトは、企業のクラウドコスト最適化（FinOps）およびソフトウェア簿価のトラッキングを行うためのモバイルダッシュボードを、PoC（概念実証）および技術ポートフォリオとして構築するものである。
  開発にあたり、以下の2つの主要な課題が存在した。
  1. **グローバルスタンダードの証明:** 開発者の持つ「アーキテクチャ設計能力」と「スケーラブルなシステム構築能力」を、世界中のトップテック企業に通用するレベルで客観的に証明すること。
  2. **厳格な機密保持 (NDA) の遵守:** 公開ポートフォリオとしてGitHub等にソースコードを公開するにあたり、過去の所属企業における実在のシステム名、業界固有のドメイン知識、あるいは実際のコストデータ等の機密情報（PII含む）が一切流出しないメカニズムを構築すること。

- **Decision (決定事項):**

  **1. アーキテクチャと技術スタックの選定**
  グローバルトレンドと高い開発者体験（DevEx）を両立するため、以下のスタックを採用する。
  - **Frontend (Mobile):** `Flutter (Dart)` - UI/UXとクロスプラットフォーム対応。
  - **Backend (API):** `FastAPI (Python)` - 実行速度と、OpenAPIを用いたSchema-Driven Developmentの実現。
  - **Architecture:** `Clean Architecture` - ドメイン駆動設計（DDD）の思想を取り入れ、ビジネスロジックをインフラ（DBや外部API）から完全に分離する。
  - **Testing:** TDD（テスト駆動開発）を原則とし、単体テスト（Pytest）およびAgentic QAをパイプラインに組み込む。

  **2. NDA遵守のための「ダミーデータ駆動開発 (Synthetic Data-Driven)」の採用**
  実データの使用を完全に排除し、バックエンドの Infra 層（`infra/generators`）において `Faker` ライブラリを用いたモックジェネレーターを構築する。
  フロントエンドは、このダミーデータをAPI経由で取得することで、「本番環境がなくともエンタープライズ品質のUIとページネーション等のスケーラビリティを証明できる」設計とする。

  **3. ドメインの完全抽象化 (White-labeling)**
  過去の所属企業や特定の業界、実在の社内システム名・アーキテクチャを推測できる用語の使用を一切禁止する。代わりに、汎用的なITインフラ用語（例: `EdgeNode`, `CloudInstance`, `CostAllocation`）をドメインエンティティとして定義する。

- **Consequences (結果と影響):**
  - **ポジティブな影響:** 
    - 開発者はコンプライアンス（NDA）を気にすることなく、安全かつ迅速にコードを公開・共有できる。
    - 「ダミーデータを用いたPoCの構築」という手法自体が、リスク管理能力の高いアーキテクトとしての評価に直結する。
    - APIスキーマが固定されるため、フロントエンドとバックエンドの並行開発（Contract-First Development）が容易になる。
  - **ネガティブな影響 (トレードオフ):**
    - バックエンド側でリアルなダミーデータを生成・管理するためのモックロジックを保守するオーバーヘッドが発生する。
